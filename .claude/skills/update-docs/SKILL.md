Ты — документатор проекта ClickHouse Analyser. Твоя задача — проанализировать изменения в коде с момента последнего обновления документации и актуализировать документы в `docs/`.

## Шаг 1: Определи scope изменений

Прочитай файл `.claude/last-docs-update` — в нём хранится хеш коммита, до которого документация уже актуализирована.

- Если файл существует и содержит хеш — используй его как начальную точку.
- Если файла нет, или передан аргумент `--all` ($ARGUMENTS содержит `--all`) — бери последние 30 коммитов как scope.

Выполни:
```bash
git log --oneline <hash>..HEAD
git diff <hash>..HEAD --stat
git diff <hash>..HEAD
```
(или `git log --oneline -30` / `git diff HEAD~30..HEAD` если маркера нет)

Внимательно изучи, что изменилось: новые фичи, багфиксы, рефакторинг, новые зависимости, изменения UI, изменения в admin-функциях.

## Шаг 2: Прочитай текущую документацию

Прочитай все файлы:
- `docs/releases.md` — история релизов
- `docs/user-guide.md` — руководство пользователя
- `docs/admin-guide.md` — руководство администратора
- `docs/architecture.md` — архитектура проекта
- `docs/development.md` — руководство разработчика
- `CLAUDE.md` — справочник для Claude (архитектурные решения, conventions)

## Шаг 3: Проанализируй, какие документы нужно обновить

Классификация:
- **Новые фичи** (новый UI, новые кнопки, новые вкладки, новая функциональность) → `releases.md` + `user-guide.md` (или `admin-guide.md` если это admin-функция)
- **Багфиксы** → `releases.md`
- **Изменения архитектуры** (новые модули, изменение паттернов, новые компоненты) → `architecture.md` + `releases.md`
- **Новые зависимости, изменения в запуске/установке** → `development.md` + `releases.md`
- **Изменения admin-функций** (подключения, мониторинг, пользователи, настройки) → `admin-guide.md` + `releases.md`

Если изменений для конкретного документа нет — НЕ трогай его.

## Шаг 4: Обнови документацию

### releases.md
- Добавь новые пункты в ВЕРХНЮЮ секцию (текущая версия).
- Если все изменения уже описаны — ничего не добавляй.
- Если номер версии изменился — создай новую секцию `## vX.Y (месяц год)`.
- Формат: подразделы по категориям (`### Категория`), bullet-списки с описанием.
- Пример формата:
  ```
  ## v0.6 (март 2026)

  ### Название категории
  - **Фича** — краткое описание, что делает и зачем
  ```

### Руководства (user-guide.md, admin-guide.md)
- Обнови существующие разделы, если функциональность изменилась.
- Добавь новые разделы для новой функциональности.
- НЕ дублируй то, что уже описано.
- Если добавляешь описание новых элементов UI — укажи где они находятся и как ими пользоваться.

### architecture.md, development.md
- Обнови при изменении структуры проекта, новых модулях, новых зависимостях, новых паттернах.

## Шаг 5: Сохрани маркер

После обновления документации запиши текущий HEAD хеш в `.claude/last-docs-update`:
```bash
git rev-parse HEAD > .claude/last-docs-update
```

## Важные правила

- **Язык**: вся документация на русском языке
- **Стиль**: сохраняй существующий стиль и формат каждого документа
- **Не переписывай**: только дополняй и актуализируй, не переделывай существующие разделы без необходимости
- **Не дублируй**: перед добавлением проверь, что это ещё не описано
- **Минимальные изменения**: если изменений для документа нет — не трогай его
- **Проверяй**: после внесения правок перечитай изменённые секции, убедись что текст читается естественно и не содержит противоречий с остальным документом
